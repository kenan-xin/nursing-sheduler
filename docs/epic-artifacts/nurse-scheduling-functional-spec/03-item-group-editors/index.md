# Item & Group Editors (People & Shift Types)

## Purpose & Scope

The People page and the Shift Types page are two instances of one shared editor
component, `ItemGroupEditorPage. Both render two side-by-side tables — an`
**items table and a groups table — plus a shared add/edit form and a set of**
per-row operations. This spec defines the behavior that is common to both pages
(shared operations, validation, drag/reorder, membership rules, checkbox
selection, duplicate labeling) and the specifics of each page (People bulk
upload; Shift Types reserved OFF item and ALL group).

- `web-frontend/src/app/people/page.tsx — People page (title="People Management", dataType=DataType.PEOPLE, adds bulk-upload header action).`
- `web-frontend/src/app/shift-types/page.tsx — Shift Types page (title="Shift Type Management", dataType=DataType.SHIFT_TYPES).`
- `web-frontend/src/components/ItemGroupEditorPage.tsx — shared editor container and all operation handlers.`

**Label derivation (**`getLabels, ItemGroupEditorPage.tsx:37-48): the singular/plural item label is derived from dataType:`
`PEOPLE → Person / People; SHIFT_TYPES → Shift Type / Shift Types;`
`DATES → Date / Dates; default → Item / Items. Group entities are`
always labeled literally `Group / Groups. All user-facing strings below are`
templated from these labels; concrete People/Shift Types renderings are shown.

**Modes (**`constants/modes.ts:21-27): NORMAL, ADDING, EDITING,`
`INLINE_EDITING, DATE_RANGE_EDITING. The editors use the first four.`

## Functional Requirements

### Page layout & header

- **FR-ED-01 The page header shows the **`title (ItemGroupEditorPage.tsx:564).`
When `instructions.length > 0, a help toggle button (icon, title="Toggle instructions")`
is shown; clicking it toggles a blue instructions panel headed `Instructions`
that renders each instruction as a bulleted line (`ItemGroupEditorPage.tsx:565-603).`
Instruction text is page-specific (People `people/page.tsx:148-155; Shift Types shift-types/page.tsx:50-57).`
- **FR-ED-02 The header renders an **`Add {itemLabel} toggle button (unless`
`itemsReadOnly) and an Add Group toggle button (unless groupsReadOnly)`
(`ItemGroupEditorPage.tsx:577-590). On People the item button reads Add Person;`
on Shift Types `Add Shift Type. A button is shown as toggled/active when`
`mode === ADDING and draft.isItem matches its kind.`
- **FR-ED-03 Two tables are rendered in a grid: the item table titled**
`{itemLabelPlural} (People / Shift Types) and the group table titled`
`{itemLabelPlural} Groups (People Groups / Shift Types Groups)`
(`ItemGroupEditorPage.tsx:626-643). On the People page only, the item table`
receives a header action: the **Upload People button (**`people/page.tsx:138-145,177).`
- **FR-ED-04 Each table has columns **`ID, a second column (Groups for the`
item table, `Members for the group table), and — unless that table is`
read-only — an `Actions column (TableColumns.tsx:180-268,282,295). The item`
ID cell is prefixed with a 1-based index (`{index+1}. {id}) because`
`showIndexPrefix is true for non-DATES item tables; the group table does not`
index-prefix (`TableColumns.tsx:118, ItemGroupEditorPage.tsx:531,548).`

### Adding an item or group

- **FR-ED-05 Clicking **`Add {itemLabel} / Add Group toggles the add form: if`
already in `ADDING mode for the same kind it cancels (closes the form);`
otherwise it enters `ADDING, resets the draft ({id:'', description:'', groups:[], members:[], isItem}),`
and clears the error (`ItemGroupEditorPage.tsx:490-508).`
- **FR-ED-06 The add/edit form (**`AddEditItemGroupForm.tsx) shows:`
  - heading `Add New {Person|Shift Type|Group} while adding (AddEditItemGroupForm.tsx:72);`
  - an ID input with placeholder `Enter {person|shift type|group} ID (:73,116-117);`
  - a description input with placeholder `Enter {person|shift type|group} description (optional) (:120);`
  - a primary action button labeled `Add while adding (:125);`
  - for an **item draft, a Groups selector; for a group draft, a Members selector (**`:127).`
- **FR-ED-07 On save (**`saveDraft, ItemGroupEditorPage.tsx:144-210): the ID and`
description are trimmed; validation runs (see Validation Rules); on success the
appropriate mutation is invoked (`addItem/addGroup), the draft is reset, mode`
returns to `NORMAL, and the error is cleared.`
- **FR-ED-08 **`addItem (useSchedulingData.ts:226-268) appends the new item`
(People items also get `history: []), adds it to every selected group, and`
re-sorts each affected group's `members into the current item order.`
`addGroup (:270-299) sorts the selected members into item order before`
storing the new group.

### Full edit (via Edit action)

- **FR-ED-09 The **`Actions column renders Edit / Duplicate / Delete buttons`
(icons; `aria-label/title = Edit, Duplicate, Delete) for`
non-auto-generated rows (`TableRowActions.tsx:29-74). Clicking Edit`
(`handleStartEditing, ItemGroupEditorPage.tsx:216-273) enters EDITING,`
loads the draft from the entity (for an item, `draft.groups = IDs of groups`
currently containing it; for a group, `draft.members = its current members),`
saves the current scroll position, and scrolls the window to top
(`behavior:'instant').`
- **FR-ED-10 Saving an edit calls **`updateItem/updateGroup`
(`useSchedulingData.ts:385-498). updateItem renames the item, updates its`
group memberships to exactly `draft.groups, and re-sorts every group's members`
into item order; `updateGroup renames the group and re-sorts its members into`
item order. Both propagate the ID change to references
(`applyReferencesForIdChange). On cancel (handleCancel) or Escape, the draft`
is discarded, mode returns to `NORMAL, and — if editing — scroll is restored`
(`ItemGroupEditorPage.tsx:350-359).`

### Inline edit (double-click)

- **FR-ED-11 Double-clicking a row's ID or description text enters inline edit**
for that field (`handleStartInlineEditing, ItemGroupEditorPage.tsx:411-438;`
wired in `TableColumns.tsx:128,136). Inline edit is offered only when the row`
is editable — `canEdit = !readOnly && !item.isAutoGenerated (TableColumns.tsx:117);`
for non-editable rows no double-click handler is attached.
- **FR-ED-12 The inline input (**`InlineEdit.tsx) auto-focuses and selects its`
contents on open (`:55-60), commits the `**trimmed value on **`Enter (guarded`
against IME composition) and on blur, and cancels on `Escape`
(`:62-74). An empty description renders the placeholder text Add description...`
(`TableColumns.tsx:139).`
- **FR-ED-13 **`handleInlineSave (ItemGroupEditorPage.tsx:440-482): for the`
`id field it validates (empty / reserved / duplicate) and calls`
`updateItem/updateGroup with the new ID only; for the description field it`
calls `updateItem/updateGroup writing back the same ID with the new`
description. On success mode returns to `NORMAL and inline state clears.`

### Delete (no confirmation)

- **FR-ED-14 Clicking Delete (**`handleDelete, ItemGroupEditorPage.tsx:291-318)`
deletes **immediately with no confirmation dialog. It first dismisses any**
active draft, then calls `deleteItem/deleteGroup. deleteItem`
(`useSchedulingData.ts:500-522) removes the item and strips it from every`
group's members; `deleteGroup (:524-541) removes the group. Both remove`
dangling references via `applyReferencesForIdDeletion.`

### Duplicate

- **FR-ED-15 Clicking Duplicate (**`handleDuplicate, ItemGroupEditorPage.tsx:320-348)`
dismisses any active draft, then calls `duplicateItem/duplicateGroup. The new`
ID is `getUniqueCopyLabel(sourceId, [...allItemIds, ...allGroupIds])`
(`useSchedulingData.ts:306-309,353-356). The clone (via structuredClone)`
keeps the source's **description verbatim — only the ID is relabeled.**
- **FR-ED-16 The duplicate is inserted immediately after the source in the**
respective list (`useSchedulingData.ts:324-328,370-374). For a duplicated`
**item, the new ID is additionally inserted into every group that contains the**
source, immediately after the source's position in that group's member list
(`:329-343).`

### Membership tags & remove-from-group

- **FR-ED-17 The item table's second column lists, as blue removable tags, the**
groups containing that item; the group table's second column lists, as gray
removable tags, that group's members (`TableColumns.tsx:200-245). Each cell is`
wrapped in `MembersCell showing a count and pluralized label`
(`{n} group|groups, {n} member|members) and a down-chevron (title="Scroll for more")`
when the tag area overflows its `max-h-28 scroll region (TableColumns.tsx:33-63,206,223).`
- **FR-ED-18 Each tag has a **`× remove button (title=Remove "{id}") unless the`
tag is read-only (`RemovableTag.tsx:105-113). A tag is read-only when either the`
item or the group is auto-generated (`TableColumns.tsx:214,239). Clicking ×`
calls `handleRemoveItemFromGroup (ItemGroupEditorPage.tsx:520-523) which`
dismisses any draft and calls `removeItemFromGroup (useSchedulingData.ts:543-563),`
removing the item from that group's members only.

### Drag-to-reorder

- **FR-ED-19 Rows can be reordered by dragging (**`DataTable.tsx:46-97). Drop`
position is computed from the cursor Y vs. the target row's vertical midpoint
(insert-before vs. insert-after), and a blue insertion indicator
(`border-t-2/border-b-2 border-blue-500) is shown at the hovered edge`
(`:64-66,88-97,140-142).`
- **FR-ED-20 Reordering is disabled (no **`onReorder passed) while`
`mode === INLINE_EDITING or the table is read-only`
(`ItemGroupEditorPage.tsx:631,640). Auto-generated rows are non-draggable: they`
carry the class `non-draggable and DataTable sets draggable=false for them`
(`ItemGroupEditorPage.tsx:632,641; DataTable.tsx:128,132).`
- **FR-ED-21 On drop, **`handleReorderItems/handleReorderGroups`
(`ItemGroupEditorPage.tsx:510-518) dismiss any active draft then call`
`reorderItems/reorderGroups. reorderItems (useSchedulingData.ts:565-581)`
additionally re-sorts every group's members into the new item order.

### Checkbox selection gesture (Groups / Members selectors)

- **FR-ED-22 The selector (**`CheckboxList.tsx) supports a click + drag`
multi-select gesture (contract documented at `CheckboxList.tsx:20-28):`
  - A plain click toggles a single checkbox on mouse-up, provided the pointer
never left the initial checkbox (`:117-125).`
  - Once the pointer leaves the initial checkbox, the gesture becomes
drag-selection: the initial checkbox is toggled at the moment of leaving
(`:106-115), and thereafter each checkbox is toggled on mouse-enter`
(`:90-95). Re-entering a checkbox toggles it again, so one checkbox may be`
toggled multiple times in one drag.
  - Mouse-up anywhere (including outside the component, via a global `mouseup`
listener) ends the gesture (`:117-139). Only the primary button`
(`button === 0) participates (:98,118). Text selection is suppressed`
(`user-select: none on body) for the duration of a drag (:67-73).`
  - Native checkbox `onChange is intentionally a no-op so all toggles flow`
through the mouse gesture logic; in `radio mode onChange toggles directly`
(`:159-164).`
- **FR-ED-23 Selecting a group in the item form stores the group ID in**
`draft.groups; selecting a member in the group form stores the item ID in`
`draft.members (ItemGroupEditorPage.tsx:390-409). Groups are not expanded to`
their members — the stored value is the group's own ID. The checkbox label text
is the entity's `id and its checked state reflects membership`
(`CheckboxList.tsx:162,166).`
- **FR-ED-24 The selectors hide auto-generated entities via **`filterAutoGenerated`
(`ItemGroupEditorPage.tsx:616, AddEditItemGroupForm.tsx:74-75,`
`keywords.ts:153-155). If no selectable items exist, the members selector shows`
`No {people|shift types} available. Please set up {people|shift types} first.`
(`AddEditItemGroupForm.tsx:76-82); if no groups exist, the groups selector shows`
`No groups available. (:95-101).`

### Keyboard (form-level)

- **FR-ED-25 While **`mode === ADDING || EDITING, a global keydown listener maps`
`Enter → save and Escape → cancel (ItemGroupEditorPage.tsx:362-379). Enter`
is ignored while an IME composition is active — `isImeCompositionKeyEvent`
returns true when `isComposing or keyCode === 229`
(`keyboardEvents.ts:24-28). The same IME guard applies to Enter inside the`
inline editor (`InlineEdit.tsx:67).`

### Auto-generated rows (OFF / ALL)

- **FR-ED-26 Auto-generated rows are rendered with a highlighted style**
(`bg-blue-50 border-blue-200) and, in the Actions column, display the literal`
text `Auto instead of action buttons (TableColumns.tsx:256-258;`
`ItemGroupEditorPage.tsx:632,641). They are non-editable (no inline double-click,`
FR-ED-11), non-draggable (FR-ED-20), and non-deletable (Delete button absent).
- **FR-ED-27 Auto-generated entities per data type (**`keywords.ts:37-141):`
  - **People: no auto-generated items; one auto-generated group **`ALL`
(description `Group containing all people).`
  - **Shift Types: one auto-generated item **`OFF (description Off shift type);`
one auto-generated group `ALL (description Group containing all shift types).`
- **FR-ED-28 The reserved keywords for a data type are exactly the IDs of its**
auto-generated items and groups (`getReservedKeywords, keywords.ts:143-147),`
matched **case-insensitively (**`isReservedKeyword, :149-151). Thus People`
reserves `ALL/all; Shift Types reserves OFF/off and ALL/all. All`
data-mutation functions also guard against reserved IDs at the hook layer,
logging an error and aborting (e.g. `useSchedulingData.ts:234-237,278-281).`

### People bulk upload (People page only)

- **FR-ED-29 The Upload People button (**`people/page.tsx:138-145) accepts`
`.txt and .csv files, with tooltip:`
`Upload a file with newline-separated people names. Existing people will be reordered to match the file order, new people will be added, and any existing people not in the file will be moved to the end.`
- **FR-ED-30 **`processPeopleFile (people/page.tsx:109-136) reads the file as`
text. If the file has no content it alerts `No content found in the uploaded file.`
Otherwise it splits on `\n, trims each line, and keeps only non-empty lines that`
do not start with `# (blank lines and # comment lines are skipped). If the`
resulting count exceeds `MAX_UPLOADED_PEOPLE (1000, people/page.tsx:31) it`
alerts `Uploaded file contains {n} people, which exceeds the maximum of 1000. Please split the file and upload fewer names at a time. and aborts.`
- **FR-ED-31 **`bulkAddPeople (people/page.tsx:50-105): if the parsed list is`
empty it alerts `No people names found in the uploaded file.. If a name repeats`
within the file it alerts `Duplicate person name "{name}" found in the uploaded list. Please remove duplicates.`
and aborts. Otherwise it rebuilds the item list as: existing people reordered to
the file's order + new names appended (created as `{id:name, description:'', history:[]})`
  - any existing people not listed appended at the end. The update is applied via
`reorderItems(DataType.PEOPLE, updatedData, updatedData.items).`
- **FR-ED-32 On success it alerts**
`Successfully uploaded {names.length} people: {existingReorderedCount} existing people reordered, {newlyAddedCount} new people added, {existingMovedToEndCount} existing people moved to end.`
(`people/page.tsx:104).`

## Validation Rules & Messages

Messages are quoted verbatim; templated tokens shown as `{...}. {itemLabel} is`
`Person (People) or Shift Type (Shift Types); group entities use the literal`
`Group.`

| --- | --- | --- | --- | --- |

Duplicate detection for V3/V6 uses `isDuplicateId(id, currentId) which compares`
against both item and group IDs, excluding the entity being edited
(`ItemGroupEditorPage.tsx:139-142). Editing the ID input clears any current error`
(`:381-384).`

### Duplicate labeling (`getUniqueCopyLabel, duplicateLabels.ts:20-43)`

- Trims the source label; strips a trailing ` copy or  copy {n} suffix`
(case-insensitive, pattern `/\s+copy(?: \d+)?$/i), then appends  copy.`
Example: `Alice → Alice copy; Alice copy → Alice copy (re-uses base,`
then dedupes → `Alice copy 2).`
- If the first candidate collides with an existing label, appends ` 2, then`
` 3, etc., until unique.`
- If the source label is empty/whitespace, uses the fallback (default `Copy).`
- In the item/group editors this generates the **new ID for a duplicated item or**
group (`useSchedulingData.ts:306,353); the fallback Copy path is only reached`
when a label is empty, which does not occur for IDs (IDs are non-empty). The
description-copy variant (`schedulingEntryDuplication.ts:34-40) applies the same`
function to descriptions elsewhere (preferences/export) and does hit the empty →
`Copy fallback.`

## Edge Cases & Quirks

- **Delete is destructive and silent — no confirmation prompt of any kind**
(FR-ED-14).
- **Full-edit vs. inline-edit duplicate messages differ: full edit says**
`...another {label} or group (V3); inline edit says ...another {label} with no`
trailing "or group" (V6).
- **`handleStartEditing`**** sets ****`EDITING`**** before the auto-generated / read-only**
**guards (**`ItemGroupEditorPage.tsx:218 runs before :221-254). In practice the`
Edit button is not rendered for auto-generated rows (they show `Auto), so this`
path is not reachable via the UI; guards that do fire only `console.error with`
`ERROR_SHOULD_NOT_HAPPEN.`
- **Duplicate copies the description verbatim (only the ID is relabeled), unlike**
the description-relabeling used for preferences/export duplication.
- **Duplicated item is inserted into groups containing the source right after the**
source's membership position, so it inherits group membership (FR-ED-16).
- **Membership is always re-sorted to item order on add, edit, and item reorder**
(FR-ED-08, FR-ED-10, FR-ED-21). The order in which members were selected is not
preserved.
- **CheckboxList re-entry toggling: dragging back over an already-visited**
checkbox toggles it again; a checkbox can flip on/off multiple times within a
single drag gesture.
- **Reserved-keyword match is case-insensitive, so **`off, all (any case) are`
rejected as IDs where those keywords apply (FR-ED-28).
- **Bulk upload bypasses the reserved-keyword and per-item add guards: it writes**
items directly via `reorderItems/updateData, so it does not surface the`
reserved-keyword validation and name matching is exact/case-sensitive
(`people/page.tsx:56,68). The success alert always reports names.length even`
though duplicates within the file abort earlier.
- **Empty-selector messaging: when there are no non-auto-generated items, the**
group-member selector shows the "set up first" message; the ALL/OFF entities are
filtered out of selectors so they never appear as selectable options.
- **`MembersCell`**** overflow indicator only appears when tag content exceeds the**
fixed-height scroll area, and is re-evaluated on window resize
(`TableColumns.tsx:37-47).`
- **Editing state warns on tab switch: **`useTabSwitchWarning is armed whenever`
`mode is ADDING, EDITING, or INLINE_EDITING (ItemGroupEditorPage.tsx:114).`

## Acceptance Criteria

- **AC-ED-01 Given the item table, when the user triggers "add item", an add**
form for that item type appears; triggering it again while that same add form is
open closes it.
- **AC-ED-02 Given the add/edit form with an empty (or whitespace-only) ID, when**
the user saves, the entity is not created/updated and the empty-ID message
(V1/V4) is shown.
- **AC-ED-03 Given an ID that matches a reserved keyword in any letter case, when**
the user saves, the entity is rejected with the reserved-keyword message
(V2/V5).
- **AC-ED-04 Given an ID already used by any existing item or group (other than**
the one being edited), when the user saves, the entity is rejected with the
duplicate message (V3 for add/full-edit, V6 for inline edit).
- **AC-ED-05 Given a valid new item with selected groups, when saved, the item is**
created and appears as a member of each selected group, and every affected
group's member list is ordered to match item order.
- **AC-ED-06 Given a valid new group with selected members, when saved, the group**
is created with its members ordered to match item order.
- **AC-ED-07 Given a double-click on an editable row's ID or description, when the**
user commits with Enter or by blurring, the trimmed value is saved; committing
with Escape discards the change.
- **AC-ED-08 Given an editable row, when the user activates Delete, the entity is**
removed immediately without any confirmation and is stripped from all groups it
belonged to (and dangling references removed).
- **AC-ED-09 Given an editable entity, when the user activates Duplicate, a copy**
is created immediately after the source with a `... copy/... copy N-style`
unique ID and the same description; a duplicated item is also added to every
group that contained the source, immediately after the source in each such group.
- **AC-ED-10 Given drag-reorder of items, when dropped, the item order changes**
accordingly and each group's members are re-ordered to match; reorder is
unavailable while inline editing and for auto-generated rows.
- **AC-ED-11 Given a membership tag that is not read-only, when the user removes**
it, the item is removed from that group only; auto-generated item/group
memberships expose no remove control.
- **AC-ED-12 Given the group/member selector, when the user clicks a single**
checkbox, exactly that one toggles; when the user presses and drags across
checkboxes, each entered checkbox toggles (including the starting one as the
pointer leaves it), and the gesture ends on mouse-up anywhere.
- **AC-ED-13 Given an item selecting a group, the stored selection is the group's**
own ID (the group is not expanded into its members).
- **AC-ED-14 Given the add/edit form is open, when the user presses Enter (with**
no active IME composition) the draft is saved, and when the user presses Escape
the draft is cancelled.
- **AC-ED-15 Given an auto-generated entity (People **`ALL; Shift Types OFF,`
`ALL), it is displayed as read-only: not inline-editable, not draggable, has no`
Edit/Duplicate/Delete controls (shows an `Auto marker), and its ID cannot be`
reused by another entity.
- **AC-ED-16 Given a **`.txt/.csv upload on the People page, names are parsed one`
per line with trimming and with blank lines and `#-comment lines skipped; a file`
over 1000 names, an empty file, an empty result set, or a file containing a
repeated name each yields its specific alert (V7–V10) and aborts.
- **AC-ED-17 Given a valid People upload, existing people are reordered to the**
file order, new names are appended as new people, existing people absent from the
file are moved to the end, and a success summary alert (V-format in FR-ED-32) is
shown.
