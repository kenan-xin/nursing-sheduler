# Save / Load & YAML

## Purpose & Scope

This artifact specifies the **Save / Load capability (the "10. Save and Load"**
tab) and every function that reads or writes the scheduling scenario as YAML:
generating the current-state YAML, downloading it, uploading/parsing a file,
copying to clipboard, editing YAML inline, the **Anonymize YAML panel, and the**
**app-version integrity check performed on every load. All behavior is**
specified **UI-agnostically at strict behavioral parity — exact strings,**
key ordering, defaults, and quirks are hard requirements.

**In scope**

- Deterministic YAML serialization of the (auto-generated-filtered) scheduling
state, including exact key order, flow-style leaf arrays, date formatting, and
the `appVersion field.`
- Download (plain and anonymized), Upload (with extension validation), Copy,
and Edit-YAML flows and all their user-visible strings.
- `loadFromYaml full-state-replacement semantics and import normalization,`
including numeric-ID coercion, date zero-padding, scalar→array wrapping,
nested-reference preservation + warnings, and required-field errors.
- The **version-mismatch integrity check (**`getVersionWarning) with its three`
cases and `confirm() dialog strings. This is load-integrity, `**in scope.**
- The **Anonymize panel (three toggles, defaults, download naming) and the**
shared anonymize transform used both here and by the Sentry attachment
helper (the transform only — Sentry transport is excluded).

**Out of scope**

- The GitHub-tag **`VersionWarningBanner (`**`components/VersionWarningBanner.tsx,`
mounted in `app/layout.tsx:77) that polls GitHub for a newer release. This is`
an **operational / ops concern, not scenario load-integrity, and is**
explicitly **NOT specified here.**
- Sentry error-event transport. Only the anonymize+serialize transform in
`utils/sentrySchedulingState.ts is documented, to pin the `*shared anonymize*
behavior.
- Local-storage persistence and undo/redo history internals (see artifact
`07-state-history-persistence); this artifact only notes that a load creates`
a new history state and persists it.

## Functional Requirements

### YAML generation (serialization)

**FR-SL-01 — State assembled for serialization, filtered, in fixed key order.**
The current-state object is built (`app/save-and-load/page.tsx:79-87) by passing`
this literal through `filterAutoGeneratedState(...) and then to`
`generateYamlFromState:`

```
{ apiVersion, description, dates, people, shiftTypes, preferences, (export?) }
```

- `filterAutoGeneratedState removes auto-generated items/groups before`
serialization, so downloaded/copied YAML never contains machine-added entries.
- `export is included `**only if **`exportData (state.export) is set`
(`...(exportData ? { export: exportData } : {}), page.tsx:86). When absent,`
no `export: key appears.`

**FR-SL-02 — ****`appVersion`**** is appended LAST.**
`generateYamlFromState spreads the state object and appends`
`appVersion: CURRENT_APP_VERSION as the final key (yamlGenerator.ts:110-113).`
It therefore always serializes after `preferences/export.`
`CURRENT_APP_VERSION is process.env.NEXT_PUBLIC_APP_VERSION || 'unknown'`
(`version.ts:23).`

**FR-SL-03 — Leaf arrays render in flow style ****`[a, b]`****.**
`replacer (yamlGenerator.ts:80-89) detects "leaf arrays" — arrays whose every`
element is `string | number | boolean | null | undefined (isLeafArray,`
`yamlGenerator.ts:34-43) — and wraps them in a CustomDump with flowLevel: 0`
so they serialize inline (e.g. `members: [Person 1, Person 2],`
`date: [01, 02]). Arrays that contain nested arrays/objects (e.g. advanced`
reference trees, `shiftTypeCoefficients pairs) are `**not leaf arrays and**
render in block style.

**FR-SL-04 — ****`Date`**** values serialize as ****`YYYY-MM-DD`****.**
`replacer converts any Date via value.toISOString().split('T')[0]`
(`yamlGenerator.ts:85-87), yielding a bare YYYY-MM-DD scalar (used for`
`dates.range.startDate / endDate).`

**FR-SL-05 — Fixed dump options.**
Serialization uses `indent: 2, lineWidth: 120, noRefs: true`
(`yamlGenerator.ts:103-108), a schema extended with the !format flow type,`
and `noCompatMode: true (yamlGenerator.ts:56, 77). Output is .trim()-ed and`
a single trailing newline `'\n' is appended (yamlGenerator.ts:115).`

**FR-SL-06 — Byte-stable / idempotent output.**
Because key order is fixed (FR-SL-01/02), leaf-array flow style is deterministic,
and `loadFromYaml re-normalizes preference/export ordering`
(`normalizePreferencesOrder, normalizeExportConfigOrder,`
`useSchedulingData.ts:960-961), serializing → loading → re-serializing the same`
scenario MUST produce byte-identical YAML.

### Download

**FR-SL-07 — Download builds a Blob and triggers a client download.**
`downloadYaml (page.tsx:93-103) creates`
`new Blob([yamlContent], { type: 'application/x-yaml' }), an object URL, a`
transient `<a download> element, clicks it, removes it, and revokes the URL.`

**FR-SL-08 — Download filename.**
Filename is
``nurse-scheduling${filenameSuffix}-${new Date().toISOString().split('T')[0]}.yaml``
(`page.tsx:98). Plain download (handleDownload) uses suffix '' →`
`nurse-scheduling-YYYY-MM-DD.yaml. Anonymized download uses suffix`
`'-anonymized' → nurse-scheduling-anonymized-YYYY-MM-DD.yaml. The date is the`
current UTC date.

### Version-mismatch integrity check

**FR-SL-19 — ****`getVersionWarning`**** has exactly three warning cases.**
`getVersionWarning(parsedData) (page.tsx:153-170) reads data.appVersion`
(`fileVersion) and returns:`

1. **Missing version — **`fileVersion falsy:`
 `"The loaded file does not contain app version information. It may have been created with an older version of the application. Current app version: ${CURRENT_APP_VERSION}"`
2. **Dirty build — **`fileVersion.endsWith('-dirty'):`
 `"Dirty app version detected.\n\nFile app version: ${fileVersion}\nCurrent app version: ${CURRENT_APP_VERSION}\n\nThis YAML was created by a development build with uncommitted changes. It may not match a reproducible application version. If nothing breaks, you can continue."`
3. **Mismatch — **`fileVersion !== CURRENT_APP_VERSION:`
 `"App version mismatch detected.\n\nFile app version: ${fileVersion}\nCurrent app version: ${CURRENT_APP_VERSION}\n\nOlder YAML may not work after breaking changes, though we try to preserve compatibility. If nothing breaks, you can continue."`
   Otherwise (`fileVersion === CURRENT_APP_VERSION) it returns null and no`
    dialog is shown. Cases are evaluated in this order (dirty is checked before
    plain mismatch). The comparison is exact string equality; it does **not**
    call `parseVersionParts.`

**FR-SL-20 — ****`parseVersionParts`**** is not used by load-integrity.**
`parseVersionParts (version.ts:58-95) parses -dirty suffixes, hash-only`
(`^[0-9a-fA-F]{7,}$), vX.Y.Z-N-gHASH, and vX.Y.Z forms into`
`{major, minor, patch, commitsAfterTag, commitId, dirty}. It underpins release`
comparison/sorting (`compareVersionsDescending) for the out-of-scope ops`
banner, **not the load-integrity check in FR-SL-19. Documented here only to**
disambiguate: the save/load version gate uses `endsWith('-dirty') + !==.`

### `loadFromYaml — full-state replacement & normalization`

**FR-SL-21 — Full-state replacement with defaults for omitted sections.**
`loadFromYaml(yamlData) (useSchedulingData.ts:771-970) builds a brand-new`
`SchedulingState (via addAutoGeneratedToState) — it `**replaces the entire**
scenario; it does not merge into current state. Missing/omitted sections are
defaulted (`useSchedulingData.ts:786-804):`

| Section | Missing/empty behavior |
| --- | --- |
| `apiVersion` | `data.apiVersion |
| `description` | `data.description |
| `dates.range` | present → `startDate/endDate each new Date(...) if set, else undefined; absent → { startDate: undefined, endDate: undefined }` |
| `dates.items / dates.groups` | ` |
| `people.items / people.groups` | ` |
| `shiftTypes` | ` |
| `preferences` | ` |
| `export` | included **only if **`data.export present (...(data.export ? { export: data.export } : {}))` |

Consequently an empty or section-omitting YAML **clears stale data for that**
section rather than retaining prior values.

**FR-SL-22 — Numeric IDs coerced to strings.**
`convertIdsToString (useSchedulingData.ts:807-817) mutates any id that is a`
`number to String(id) for dates.items, dates.groups, people.items,`
`people.groups, shiftTypes.items, shiftTypes.groups.`

**FR-SL-23 — Reference-ID stringification; date IDs zero-padded.**
`stringifyReferenceIds (useSchedulingData.ts:818-826) recursively walks`
arrays; a numeric leaf becomes `value.toString().padStart(2, '0') when it is a`
**date reference (**`isDate = true) or String(value) otherwise. So a date`
reference `1 imports as "01" (two-digit zero-pad), while a non-date numeric`
reference `1 imports as "1".`

**FR-SL-24 — Scalar reference wrapped into a single-element array.**
`normalizeReferenceIdsForImport (useSchedulingData.ts:827-841): null/`
`undefined pass through unchanged; otherwise the value is stringified and, if`
not already an array, wrapped as `[value]. This lets scalar YAML shorthands`
(e.g. `person: P1) normalize to person: [P1].`

**FR-SL-25 — Nested reference trees preserved + flagged.**
When a normalized value `hasNestedReferenceIds (an array containing any array —`
`referenceIds.ts:22-24), it is `**kept as-is and a warning is pushed:**
``${path} uses advanced backend reference syntax. It was preserved and may not be editable in the web UI without replacing it.` (useSchedulingData.ts:835-839).`

**FR-SL-26 — Required export reference fields throw when null.**
`normalizeRequiredReferenceIdsForImport (useSchedulingData.ts:842-847) throws`
``${path} is required` if the value is null/undefined. Applied to`
`export.extraColumns[i].countShiftTypes, export.extraColumns[i].countDates,`
`export.extraRows[i].countShiftTypes, export.extraRows[i].countPeople`
(`useSchedulingData.ts:933-959). Such a throw surfaces as the upload/edit`
parse-error path (FR-SL-14 / FR-SL-18).

**FR-SL-27 — Missing ****`date`**** on requirement/succession defaults to ****`[ALL]`****.**
For each preference (`useSchedulingData.ts:853-860): a SHIFT_TYPE_REQUIREMENT`
or `SHIFT_TYPE_SUCCESSIONS with no date property gets date = [ALL]`
(`ALL = 'ALL'), since the backend treats an omitted date as "all dates" but the`
frontend needs an explicit selector.

**FR-SL-28 — Per-preference field normalization.**
The importer normalizes these fields with `normalizeReferenceIdsForImport (date`
fields flagged `isDate): shiftType, qualifiedPeople, date (date),`
`person, pattern, countDates (date), countShiftTypes, people1,`
`people2, shiftTypes (useSchedulingData.ts:863-904). Coefficient pair lists`
are stringified on the id side: `shiftTypeCoefficients (requirement) and`
`countShiftTypeCoefficients (shift-count) each map to [String(id), coefficient]`
(`useSchedulingData.ts:866-871, 890-895).`

**FR-SL-29 — Unsupported-shape warnings for backend-only preference forms.**
`addUnsupportedFrontendShapeWarning(path) pushes`
``${path} uses backend-compatible syntax that is outside the web UI editing subset. It was preserved and may be replaced if edited in the web UI.` (useSchedulingData.ts:848-852). It fires when`
(`useSchedulingData.ts:906-920): a SHIFT_REQUEST has person.length !== 1`
(`.person) or shiftType.length !== 1 (.shiftType); or a SHIFT_COUNT has an`
array `expression (.expression) or array target (.target).`

**FR-SL-30 — Export section normalization.**
`export.formatting[]: people, dates (date), shiftTypes normalized`
(`useSchedulingData.ts:922-932). export.extraColumns[] and`
`export.extraRows[]: required fields per FR-SL-26, plus`
`extraColumns[].countShiftTypeCoefficients stringified to [String(id), coef]`
(`useSchedulingData.ts:933-959).`

**FR-SL-31 — Deduped warnings surfaced via banner; ordering normalized; new history state.**
Collected warnings are deduplicated with `[...new Set(importWarnings)] and set`
as `yamlImportWarnings (useSchedulingData.ts:962). Preference and export`
ordering are re-normalized (`normalizePreferencesOrder,`
`normalizeExportConfigOrder, useSchedulingData.ts:960-961). The new state is`

`useSchedulingData.ts:964-969).`

**FR-SL-32 — Import-warnings banner.**
When `yamlImportWarnings.length > 0, a banner (YamlImportWarningsBanner,`

## Edge Cases & Quirks

- **`appVersion`**** last, always present. Even a scenario with no export still**
serializes `appVersion as the final key. The value can be the literal`
`"unknown" when NEXT_PUBLIC_APP_VERSION is unset (FR-SL-02). [incidental`
quirk: `"unknown" compared exactly against the file's version.]`
- **`export`**** presence is asymmetric. On serialize it depends on **`state.export`
being truthy (FR-SL-01); on import it is included only if the YAML has an
`export key (FR-SL-21). A round-trip of a scenario with no export omits the`
key both ways.
- **Date-reference zero-pad is one-sided. Only date references pad to 2 digits**
(`"01"); shift-type/people numeric references become plain String ("1")`
(FR-SL-23). This is why the day-of-month IDs in generated calendars are
two-char strings.
- **Scalar→array wrap changes shape. Importing **`person: P1 yields`
`person: [P1]; a subsequent serialize emits the array flow form (FR-SL-24,`
FR-SL-03). Round-trips from web-UI-generated YAML are already arrays, so
byte-stable (FR-SL-06); hand-authored scalars normalize on first import.
- **Missing ****`date`**** → ****`[ALL]`**** only for requirement/succession. Other preference**
types are untouched (FR-SL-27).
- **Nested reference trees survive but are flagged. They keep block-style**
serialization (not leaf arrays) and raise the advanced-syntax warning
(FR-SL-25, FR-SL-03).
- **Warnings are deduped by exact string. Two preferences producing the same**
`path-less message collapse; distinct paths remain separate`
(FR-SL-31, `[...new Set(...)]).`
- **`filterAutoGeneratedState`**** before serialize but not the raw import. The**
serialized/downloaded YAML excludes auto-generated entries; on import
`addAutoGeneratedToState re-adds them (FR-SL-01, FR-SL-21).`
- **Version check is string equality, not semver. A **`dirty suffix is caught`
before plain inequality; `parseVersionParts/semver ordering is not consulted`
here (FR-SL-19, FR-SL-20). [incidental quirk]
- **Copy failure is silent to the user. Only **`console.error; no alert`
(FR-SL-09).
- **Anonymize never mutates live state. Both scatter and anonymize operate on**
copies; the on-screen scenario and YAML are unchanged after an anonymized
download (FR-SL-35).
- **Anonymized ID collision avoidance. Retained (non-anonymized) IDs seed**
`usedIds, so P/G sequences skip any pre-existing matching id (FR-SL-36).`
- **`removeDescriptions`**** differs by caller. Save/Load anonymize keeps**
descriptions; the Sentry helper strips them (FR-SL-36, FR-SL-39).
- **Same-file re-upload works. The file input value resets to **`'' after a`
pick so choosing the identical file fires `onChange again (FR-SL-11).`

## Acceptance Criteria

*(UI-agnostic given/when/then; seed the parity suite.)*

- **AC-SL-01 — Given a scenario, when the current-state YAML is generated, then**
top-level keys appear in order `apiVersion, description, dates, people, shiftTypes, preferences, [export], appVersion, with appVersion last and`
`export present iff the scenario has an export config.`
- **AC-SL-02 — Given a scenario whose **`appVersion equals the current app`
version, when it is serialized then loaded then re-serialized, then the output
is byte-identical (including the single trailing newline).
- **AC-SL-03 — Given a field that is an array of only primitives, when**
serialized, then it renders in flow style `[a, b]; given an array containing a`
nested array, then it renders in block style.
- **AC-SL-04 — Given a **`Date-typed value, when serialized, then it appears as a`
bare `YYYY-MM-DD scalar.`
- **AC-SL-05 — Given serialization, then indentation is 2 spaces and no YAML**
anchors/aliases (`noRefs) appear.`
- **AC-SL-06 — When a plain download is triggered, then a file named**
`nurse-scheduling-<UTC-date>.yaml with blob type application/x-yaml is`

`nurse-scheduling-anonymized-<UTC-date>.yaml.`

- **AC-SL-08 — Given a file whose extension is not **`.yaml/.yml, when`
selected or dropped, then it is rejected with

occurs.
- **AC-SL-09 — Given a drop with no file, then **`No file was dropped.; given an`

- **AC-SL-10 — Given a valid YAML upload with a matching version, then the state**
is fully replaced and `YAML file loaded successfully! is shown; given invalid`

with no state change.
- **AC-SL-12 — Given Edit-YAML Save with unparseable YAML, then the inline error**
shows the parser message (or `Invalid YAML format) and the editor stays open;`
editing the textarea clears the error.
- **AC-SL-13 — Given a loaded YAML that omits a section (or is empty), then that**

- **AC-SL-14 — Given item/group **`ids that are numbers, when loaded, then they`

- **AC-SL-16 — Given a **`SHIFT_TYPE_REQUIREMENT or SHIFT_TYPE_SUCCESSIONS`

- **AC-SL-18 — Given a **`SHIFT_REQUEST with person or shiftType length ≠ 1,`

- **AC-SL-19 — Given an **`export.extraColumns/extraRows required reference`

- **AC-SL-20 — Given multiple identical import warnings, when a load completes,**

- **AC-SL-21 — Given the Anonymize panel on first render, then "Replace people**

## Appendix — Generated YAML output schema (C1 contract mirror)

Representative shape of `generateYamlFromState output. Leaf arrays render inline`
(`[a, b]); Dates render as YYYY-MM-DD; appVersion is always last; export`
appears only when present. See C1 for the authoritative per-field contract.

```yaml
apiVersion: alpha                 # data.apiVersion || 'alpha'
description: ""                   # free text
dates:
  range:
    startDate: 2026-01-01         # Date -> YYYY-MM-DD (may be omitted/undefined)
    endDate: 2026-01-31
  items:
    - id: "01"                    # date item IDs are zero-padded strings
      description: ""
  groups:
    - id: WORKDAY
      members: ["01", "02"]       # leaf array -> flow style
      description: ""
people:
  items:
    - id: Person 1
      description: ""
      history: []
  groups:
    - id: Group 1
      members: [Person 1, Person 2]
      description: ""
shiftTypes:
  items:
    - id: D
      description: Day (All Levels)
  groups:
    - id: Day
      members: [D, D+]
      description: All day shift types
preferences:
  - type: shift request           # scalars normalized to single-element arrays on import
    person: [Person 1]
    date: ["01", "02"]
    shiftType: [D]
    weight: 1
  - type: shift type requirement
    date: [ALL]                   # omitted date imported as [ALL]
    shiftType: [D]
    qualifiedPeople: [Person 1]
    shiftTypeCoefficients:        # pair list -> NOT a leaf array (block style)
      - [D, 1]
export:                           # present only when state.export is set
  formatting:
    - people: [Person 1]
  extraColumns:
    - countShiftTypes: [D]        # required on import (throws "<path> is required" if null)
      countDates: ["01"]
  extraRows:
    - countShiftTypes: [D]        # required
      countPeople: [Person 1]    # required
appVersion: v1.2.3                # ALWAYS last; CURRENT_APP_VERSION (may be "unknown")
```
