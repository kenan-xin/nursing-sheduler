# Data Model & Entities

## Purpose & Scope

This spec defines the canonical in-memory/persisted data model for the nurse-scheduling application: the entities that describe a scheduling problem (people, shift types, dates, groups, preferences), how they reference one another, and the invariants the app maintains. It is the authoritative catalog of every type, field, default value, reserved keyword, and auto-generated entity, plus the canonical ordering/normalization rules applied before state is saved or exported.

Scope is the data model only. It describes WHAT data the app holds and the rules that govern that data, not how any of it is rendered. Date-item ID generation and the mechanics of injecting/stripping auto-generated entities are summarized here and specified in full in spec 02.

Source of truth for this spec:

- `web-frontend/src/types/scheduling.ts (all types/fields)`
- `web-frontend/src/hooks/schedulingState.ts (SchedulingState shape + default seed)`
- `web-frontend/src/utils/keywords.ts (reserved keywords, auto-generated catalogs, API_VERSION)`
- `web-frontend/src/hooks/schedulingGeneratedData.ts (auto-generation + strip logic)`
- `web-frontend/src/utils/entityOrdering.ts (canonical entity ordering)`
- `web-frontend/src/utils/referenceIds.ts (flat vs nested reference trees)`
- `web-frontend/src/hooks/schedulingPreferenceOrdering.ts (preference normalization/ordering)`
- `web-frontend/src/hooks/schedulingConstants.ts (storage + sentinel constants)`

## Functional Requirements

### Root state container

**FR-DM-01 — The application holds one root **`SchedulingState object with a fixed set of top-level fields. Per **`**`schedulingState.ts:24-32, SchedulingState has: apiVersion: string | number; description: string; dates: { range: DateRange, items: Item[]; groups: Group[] }; people: { items: Item[]; groups: Group[] }; shiftTypes: { items: Item[]; groups: Group[] }; preferences: Preference[]; and optional export?: ExportConfig. The three data domains are dates, people, and shiftTypes; only dates carries an additional range field. export is optional and may be absent.`

<user_quoted_section>Frontend runtime vs persisted-shape note: in the running app, dates.items is always populated at runtime by addAutoGeneratedToState (one Item per day in the range; spec 02 FR-DC-11/12). However, when state is persisted to localStorage (schedulingStorage.ts:98-135), dates.items is explicitly set to undefined for both the current state and every history entry — dates.items is not stored; it is regenerated on hydration. Spec 02 and the persistence artifact document this; the catalog reuses it.</user_quoted_section>

<user_quoted_section>People bulk-upload shape quirk: when a user uploads a people list via processPeopleFile / bulkAddPeople (app/people/page.tsx:50-78, 94-104), the upload path validates for upload-list-internal duplicates and exact-match-existing-people-items but does not run the normal addItem validation (app/people/page.tsx:139-159 and useSchedulingData.ts:386-449). In particular, bulk upload can create people item ids that collide with existing people group ids (e.g. uploading a line Group 1 while a Group 1 group already exists) and bypasses the reserved-keyword check. The spec's normal editor invariant "item ids and group ids share one namespace" (FR-DM-08) and reserved-keyword guard are enforced by the normal editor only, not by bulk upload. The cascade then treats the new id as a normal item reference target.</user_quoted_section>

**FR-DM-02 — Each data domain is keyed by the **`DataType enum with exactly three members. Per **`**`scheduling.ts:145-149, DataType = DATES = 'dates', PEOPLE = 'people', SHIFT_TYPES = 'shiftTypes'. These string values match the corresponding SchedulingState field names and are the keys of the auto-generated catalogs (FR-DM-11/12).`

**FR-DM-03 — **`apiVersion is the string alpha by default. Per **`**`keywords.ts:35, API_VERSION = 'alpha'. createDefaultState sets apiVersion: API_VERSION (schedulingState.ts:92). The type permits string | number, but the seeded and current value is the string alpha.`

### Item and Group entities (IDs-as-labels model)

**FR-DM-04 — An **`Item is an identified entity whose id is also its human-facing label. Per **`**`scheduling.ts:115-120, Item = { id: string; description: string; history?: string[]; isAutoGenerated?: boolean }. There is no separate display-name field; the id string is the label used everywhere the entity is referenced. description is a free-text annotation (required by type; seeded as '' for people, as descriptive text for shift types — see FR-DM-15/16). history and isAutoGenerated are optional.`

**FR-DM-05 — A **`Group is a named collection of item IDs within the same domain. Per **`**`scheduling.ts:122-127, Group = { id: string; members: string[]; description: string; isAutoGenerated?: boolean }. members is an array of item id strings (comment: // Array of item IDs). Groups reference items by string ID, never by object. A group's id shares the same ID-as-label namespace as items (FR-DM-08).`

**FR-DM-06 — Only people items carry assignment **`history. **`**`history?: string[] is // Array of shift type IDs representing historical assignments for people (scheduling.ts:118). In the default seed, every person item has history: [] (schedulingState.ts:39), and shift-type/date items are seeded without a history field. Each history entry is a shift-type ID string (or a reserved keyword such as OFF); ordering of the array is positional (most recent assignment history preceding the schedule).`

**FR-DM-07 — All cross-entity references are stored as ID strings, not object references. Group membership (**`Group.members), person history (Item.history), and every preference scope field (person/date/shiftType/etc.) hold arrays of id strings. A reference resolves against the item/group id values of the matching domain. Because IDs are labels, renaming an entity is a rename of the referenced string across the model.`

**FR-DM-08 — Within a domain, item IDs and group IDs share one namespace and are ordered items-first, then groups. Per **`entityOrdering.ts:24-26, getOrderedEntries returns [...data.items, ...data.groups] — the canonical ordering for a domain is all items in array order followed by all groups in array order. This ordering is the basis for normalization (FR-DM-20).`

### Reference-ID trees (flat vs nested)

**FR-DM-09 — A reference-ID value is either a flat list of IDs or a nested tree of ID lists. Per **`referenceIds.ts:20, ReferenceIdTree = string | ReferenceIdTree[]. hasNestedReferenceIds(value) returns true only when value is an array that itself contains at least one array (referenceIds.ts:22-24). Nested trees express grouped/structured selections (e.g. an ordered pattern-of-groups); flat lists express a simple set of IDs. mapReferenceIdTree recursively maps every leaf ID; filterReferenceIdTree recursively drops leaves failing a predicate and prunes emptied sub-arrays (referenceIds.ts:26-44).`

**FR-DM-10 — Nested reference structure is preserved (never flattened or reordered) during normalization; only flat reference lists are sorted. Per **`schedulingPreferenceOrdering.ts:42-48, sortFlatOrPreserveNestedReferenceIds returns the value unchanged when hasNestedReferenceIds is true, otherwise sorts it by entity order. This applies to shiftType on shift-type-requirement preferences and to people1/people2/shiftTypes on shift-affinity preferences (FR-DM-20). [incidental quirk] Whether a field is treated as nested is decided per-value at normalization time, not by static type — a field typed string[] may at runtime hold a nested tree and is then preserved as-is.`

### Reserved keywords & auto-generated entities

**FR-DM-11 — Auto-generated ITEMS are defined per domain; only shift types have one (****`OFF*). Per **`***`keywords.ts:37-46, AUTO_GENERATED_ITEMS maps PEOPLE -> [], DATES -> [], and SHIFT_TYPES -> [{ id: 'OFF', description: 'Off shift type' }]. OFF (constant keywords.ts:23) is the reserved shift-type item representing an off/no-shift assignment and is available for history entries and preference references without being user-defined.`

**FR-DM-12 — Auto-generated GROUPS are defined per domain; every domain gets **`ALL, and dates get weekday/weekend/day-of-week groups. Per **`**`keywords.ts:48-141, AUTO_GENERATED_GROUPS maps:`

- `PEOPLE -> [ ALL ] with description Group containing all people; members = every people item id.`
- `SHIFT_TYPES -> [ ALL ] with description Group containing all shift types; members = every shift-type item id.`
- `DATES -> [ ALL, WEEKDAY, WEEKEND, SUNDAY, MONDAY, TUESDAY, WEDNESDAY, THURSDAY, FRIDAY, SATURDAY ].`

Date group descriptions are exactly: `Group containing all dates (ALL), Group containing all weekdays (WEEKDAY), Group containing all weekends (WEEKEND), Group containing all Sundays (SUNDAY), Group containing all Mondays (MONDAY), Group containing all Tuesdays (TUESDAY), Group containing all Wednesdays (WEDNESDAY), Group containing all Thursdays (THURSDAY), Group containing all Fridays (FRIDAY), Group containing all Saturdays (SATURDAY). Reserved-keyword string constants: OFF, ALL, WEEKDAY, WEEKEND, MONDAY, TUESDAY, WEDNESDAY, THURSDAY, FRIDAY, SATURDAY, SUNDAY (keywords.ts:23-33).`

**FR-DM-13 — Date-group membership is data-driven by UTC weekday and requires a start date (except ****`ALL`****). Per ****`keywords.ts:70-139, each of the weekday-derived date groups (WEEKDAY, WEEKEND, SUNDAY, MONDAY, TUESDAY, WEDNESDAY, THURSDAY, FRIDAY, SATURDAY) returns [] unless dateRange && dateRange.startDate is truthy; otherwise it filters date items by dateStrToDate(item.id, dateRange).getUTCDay() where 0 = Sunday … 6 = Saturday. WEEKDAY = days 1–5 inclusive; WEEKEND = days 0 or 6; each single-day group = its exact getUTCDay() value (SUNDAY=0, MONDAY=1, …, SATURDAY=6). `****`ALL`***** is the exception: its generator is ***`(items) => items.map(item => item.id) and does **not consult **dateRange (keywords.ts:63-68). Under current strict parity, the supported date ALL always contains every available date item id, even when range.startDate is undefined (and even when the weekday-derived groups are all empty). ALL groups (all domains) otherwise simply map every filtered item to its id (keywords.ts:53,60,67). Generation detail (date-ID formatting, injection timing) is specified in spec 02.`

**FR-DM-14 — Reserved keywords are matched case-insensitively, and auto-generated entities are stripped before persist/export. **`getReservedKeywords(dataType) concatenates auto item IDs then auto group IDs (keywords.ts:143-147). isReservedKeyword compares via .toLowerCase() on both sides (keywords.ts:149-151) — reservation is case-insensitive, so all, off, weekday, etc. collide with reserved IDs regardless of case. filterAutoGenerated(items) removes every entry with truthy isAutoGenerated (keywords.ts:153-155), and filterAutoGeneratedState(state) applies this to all six item/group arrays across the three domains (schedulingGeneratedData.ts:98-116), preserving dates.range. Auto-generated entities exist only in the working (frontend-normalized) state and are stripped before the state is saved/exported. [incidental quirk] replaceDateGroups also matches replacement group IDs case-insensitively via .toLowerCase() (schedulingGeneratedData.ts:153-155).`

### Preference catalog

**FR-DM-19 — There are exactly seven preference variants, discriminated by a string **`type, unioned as Preference. Per **`**`scheduling.ts:161-247, every preference extends BasePreference { type: string; description?: string }. Type-discriminator constants (scheduling.ts:151-157): AT_MOST_ONE_SHIFT_PER_DAY = 'at most one shift per day', SHIFT_TYPE_REQUIREMENT = 'shift type requirement', SHIFT_REQUEST = 'shift request', SHIFT_TYPE_SUCCESSIONS = 'shift type successions', SHIFT_COUNT = 'shift count', SHIFT_AFFINITY = 'shift affinity', SHIFT_TYPE_COVERING = 'shift type covering'. Fields per variant:`

- **`AtMostOneShiftPerDayPreference (`**`type: 'at most one shift per day'): no fields beyond type and optional description.`
- **`ShiftTypeRequirementsPreference (`**`type: 'shift type requirement'): shiftType: string[]; shiftTypeCoefficients?: ShiftCountTypeCoefficient[]; requiredNumPeople: number; qualifiedPeople: string[]; preferredNumPeople?: number; date: string[]; weight: number. - **`ShiftRequestPreference (`**`type: 'shift request'): person: string[] (// Should only contain one person); date: string[] (// Can contain multiple dates); shiftType: string[] (// Should only contain one shift type); weight: number.`
- **`ShiftTypeSuccessionsPreference (`**`type: 'shift type successions'): person: string[]; pattern: string[] (// List of shift type IDs that are in succession); date: string[] (backend permits null/missing = all dates; frontend uses ALL); weight: number.`
- **`ShiftCountPreference (`**`type: 'shift count'): person: string[]; countDates: string[]; countShiftTypes: string[]; countShiftTypeCoefficients?: ShiftCountTypeCoefficient[]; expression: typeof SUPPORTED_EXPRESSIONS[number]; target: number; weight: number.`
- **`ShiftAffinityPreference (`**`type: 'shift affinity'): date: string[]; people1: string[]; people2: string[]; shiftTypes: string[]; weight: number.`
- **`ShiftTypeCoveringPreference (`**`type: 'shift type covering', scheduling.ts:229-237): date?: string[] (optional; `**current backend treats missing/****`null`****/********`[]`****** as NO dates / no-op — see C3 CON-SEM-07; to target "all dates" the frontend must emit ****`date: [ALL] or an explicit list of date ids); preceptors: (string | string[])[] (nested reference tree of people ids/groups); preceptees: (string | string[])[] (nested reference tree of people ids/groups); shiftTypes: (string | string[])[] (nested reference tree of shift-type ids/groups, including OFF and ALL); weight: number. Supporting types: `ShiftCountTypeCoefficient = [string, number] (scheduling.ts:159) — an [entityId, coefficient] pair; SUPPORTED_EXPRESSIONS = ['|x - T|^2', 'x >= T', 'x <= T', 'x > T', 'x < T', 'x = T'] as const (scheduling.ts:158) is the closed set of shift-count expressions where x is the counted value and T the target.`

**FR-DM-19a — The implicit-all normalization applies specifically to **`qualifiedPeople. Per **`**`schedulingPreferenceOrdering.ts:26-40, normalizeQualifiedPeopleForFrontend maps null/undefined -> [ALL], wraps a non-array scalar as [String(value)], and passes arrays through. ALL = 'ALL' (keywords.ts:24). This runs during preference normalization (FR-DM-20).`

### Canonical ordering & normalization

**FR-DM-20 — Preferences are normalized to a stable order before save/export. Per **`schedulingPreferenceOrdering.ts:133-145, normalizePreferencesOrder: (1) normalizes each preference's ID fields via normalizePreferenceOrder; (2) sorts the shift request preferences by entity order via sortShiftRequestsByEntityOrder; (3) concatenates non-request preferences then sorted requests; (4) groups all preferences by type via sortPreferencesByType. sortPreferencesByType orders by the fixed type list [at most one shift per day, shift type requirement, shift request, shift type successions, shift count, shift affinity, shift type covering] (schedulingPreferenceOrdering.ts:112-114). The trailing shift type covering ensures covering rules sort to the end of the list rather than to index -1 (which would push them to the start). sortShiftRequestsByEntityOrder sorts requests by person entity order, then shift-type entity order, then ascending weight (schedulingPreferenceOrdering.ts:117-129).`

**FR-DM-21 — Per-preference ID fields are sorted by their domain's canonical entity order. Per **`schedulingPreferenceOrdering.ts:50-110, normalizePreferenceOrder sorts, using getOrderedEntries (items-then-groups) for the relevant domain: for shift type requirement — shiftType (flat-only), shiftTypeCoefficients (by first id), qualifiedPeople (after implicit-all normalization), date; for shift request — person, date, shiftType; for shift type successions — person, date (note: pattern is intentionally left unsorted to preserve succession order); for shift count — person, countDates, countShiftTypes, countShiftTypeCoefficients (by first id); for shift affinity — date, and people1/people2/shiftTypes (flat-only, nested preserved); for shift type covering — date (flat-only, sorted; preserved as undefined when missing), while the nested preceptors/preceptees/shiftTypes trees are passed through unchanged (matching the shift-affinity convention: nested trees encode cross-product semantics that the canonical entity order would scramble). The covering normalize branch is at schedulingPreferenceOrdering.ts:98-110. Any preference type not matched is returned unchanged.`

**FR-DM-22 — Sorting helpers are null-safe and place unknown IDs last. Per **`entityOrdering.ts:28-62: sortIdsByEntryOrder returns [] for undefined/null, wraps a non-array as [String(ids)], and stable-sorts known IDs by index while unknown IDs get Number.MAX_SAFE_INTEGER (sorted to the end). compareFirstIdByEntryOrder compares only the first element of each list. sortPairsByFirstIdEntryOrder sorts [id, value] pairs by the id's entry order and passes undefined through unchanged.`

**FR-DM-23 — Frontend normalization injects auto-generated entities and orders preferences/export. Per **`schedulingGeneratedData.ts:139-146, normalizeStateForFrontend = addAutoGeneratedToState (strip then re-append auto items/groups per domain, schedulingGeneratedData.ts:118-137) followed by normalizePreferencesOrder and normalizeExportConfigOrder. This is the inverse of the persist/export path, which strips auto-generated entities (FR-DM-14). Injection detail is in spec 02.`

## Validation Rules & Messages

The data-model layer (the files in scope) enforces structural rules through types and normalization but does NOT itself emit user-facing validation message strings — no string literals in these files are surfaced as validation errors. The following are the enforced structural rules; user-facing message text lives in the editing/UI specs.

| Condition | Result / message | Source |
| --- | --- | --- |
| An entity `id case-insensitively equals a reserved keyword for its domain` | Treated as reserved (`isReservedKeyword -> true); no message emitted at this layer` | `keywords.ts:149-151` |
| A group/item has truthy `isAutoGenerated` | Excluded on persist/export by `filterAutoGenerated (silent)` | `keywords.ts:153-155; schedulingGeneratedData.ts:98-116` |
| `qualifiedPeople is null/undefined on a shift-type-requirement preference` | Normalized to `[ALL] (silent)` | `schedulingPreferenceOrdering.ts:38` |
| An ID in a preference field is not found among ordered entries | Sorted to the end via `Number.MAX_SAFE_INTEGER; retained, not removed` | `entityOrdering.ts:34-36` |
| A reference-field value is `undefined/null during sort` | Coerced to `[] (silent)` | `entityOrdering.ts:29` |
| A reference-field value is a non-array scalar during sort | Coerced to `[String(value)] (silent)` | `entityOrdering.ts:30, schedulingPreferenceOrdering.ts:39` |

## Edge Cases & Quirks

- **`WORKDAY`****/****`NON-WORKDAY`****/****`PH`****** are not reserved and not stripped. Despite being uppercase like reserved date keywords, they are seeded as ordinary user groups (no ****`isAutoGenerated), so they persist and are user-editable and initially empty (schedulingState.ts:78-93; schedulingConstants.ts:28-30).`
- **Case-insensitive reservation and replacement. **`isReservedKeyword and replaceDateGroups both compare via .toLowerCase(), so all, Off, weekday, etc. collide with reserved IDs and date-group replacement matches ignore case (keywords.ts:149-151; schedulingGeneratedData.ts:153-155).`
- **Date groups are empty without a start date. Every date auto-group's **`generateMembers returns [] when dateRange.startDate is falsy, including ALL for dates when there are no date items (keywords.ts:70-139).`
- **Nested reference trees survive normalization unchanged. Fields typed **`string[] may hold nested arrays at runtime; when nested, they are neither sorted nor flattened (schedulingPreferenceOrdering.ts:42-48), while flat lists are sorted.`
- **`pattern** (shift type successions) is deliberately not sorted. Only **`**`person and date are normalized for that preference; succession order in pattern is meaningful and preserved (schedulingPreferenceOrdering.ts:73-79).`
- **Unknown IDs are retained, not pruned. Sorting pushes unresolved IDs to the end rather than dropping them (**`entityOrdering.ts:34-36); pruning only happens for auto-generated stripping and explicit filterReferenceIdTree predicate use.`
- **Single-element array fields. **`shift request person/shiftType are typed string[] but by contract hold exactly one element (scheduling.ts:190,192); compareFirstIdByEntryOrder relies on this by comparing only [0].`
- **Explicit-********`ALL`****** vs implicit-null asymmetry. Frontend state stores ****`[ALL] for all-people/all-dates scopes while the backend also accepts null/missing; normalization converts inbound implicit forms to explicit ALL (scheduling.ts:176-183,200-201; schedulingPreferenceOrdering.ts:32-40).`
- **`OFF** is the only auto-generated item. People and dates have no auto-generated items; only shift types get **`**`OFF (keywords.ts:37-46).`
- **Infinity weights need placeholders. Because **`JSON.stringify(Infinity) === null, weights of ±Infinity are represented with __INFINITY__/__NEGATIVE_INFINITY__ for storage round-trips (schedulingConstants.ts:23-27).`

## Acceptance Criteria

- **AC-DM-01 — Given a freshly created default state, when its contents are inspected, then **`apiVersion equals alpha, description equals '', export is absent, and preferences is exactly one at most one shift per day preference.`
- **AC-DM-02 — Given a freshly created default state, when people are inspected, then there are 10 items **`Person 1…Person 10 each with description: '' and history: [], and 5 groups Group 1…Group 5 with the exact membership listed in FR-DM-15.`
- **AC-DM-03 — Given a freshly created default state, when shift types are inspected, then there are 9 items with the exact id->description pairs in FR-DM-16 and 4 groups **`Day/Evening/Night/Administrative with the listed members and descriptions.`
- **AC-DM-04 — Given a freshly created default state, when dates are inspected, then **`range.startDate and range.endDate are both undefined, items is empty, and groups is exactly WORKDAY (description 'Workdays (weekdays excluding public holidays)'), NON-WORKDAY (description 'Non-Work Days (public holidays and weekends)'), and PH (description 'Public Holidays'), all with empty members and none flagged auto-generated.`
- **AC-DM-05 — Given any state, when auto-generation runs, then a single **`ALL group is present for each of people/shiftTypes/dates, an OFF item is present for shift types only, and dates additionally have WEEKDAY, WEEKEND, SUNDAY…SATURDAY groups — all flagged isAutoGenerated: true.`
- **AC-DM-06 — Given a date domain with a defined start date and date items, when date auto-groups are generated, then **`WEEKDAY contains exactly the items whose UTC day-of-week is 1–5, WEEKEND those with day 0 or 6, and each single-day group those matching its specific getUTCDay() value.`
- **AC-DM-07 — Given a date domain whose **`range.startDate is undefined, when`
date auto-groups are generated, then `WEEKDAY, WEEKEND, and the seven`
weekday-name date auto-groups have empty members. The date `ALL`
group is the **exception and contains every available date item id**
(it does not consult `dateRange); see FR-DM-13.`
- **AC-DM-08 — Given a state being prepared for persist/export, when auto-generated stripping runs, then no entity flagged **`isAutoGenerated remains in any item or group array, while dates.range, all non-auto entities, and WORKDAY/NON-WORKDAY/PH are retained.`
- **AC-DM-09 — Given a candidate entity id equal to a reserved keyword ignoring case (e.g. **`all, Off, weekday), when checked, then it is reported reserved for its domain.`
- **AC-DM-10 — Given a shift-type-requirement preference whose **`qualifiedPeople is null or missing, when normalized for the frontend, then qualifiedPeople becomes [ALL].`
- **AC-DM-11 — Given a set of preferences, when normalized, then they are grouped in the order **`at most one shift per day → shift type requirement → shift request → shift type successions → shift count → shift affinity → shift type covering (trailing position), and shift requests within their group are ordered by person, then shift type, then ascending weight.`
- **AC-DM-12 — Given a preference reference field containing a nested array, when normalized, then its structure is preserved unchanged; given a flat reference field, then its IDs are reordered to match items-then-groups entity order with unknown IDs placed last.**
- **AC-DM-13 — Given a **`shift type successions preference, when normalized, then person and date are reordered but pattern order is left intact.`
- **AC-DM-14 — (Restricted to preference/reference fields.) Given a**
preference reference field (e.g. `person, shiftType, pattern,`
`preceptors, preceptees, shiftTypes, qualifiedPeople,`
`countShiftTypes, countDates, people1, people2, shiftTypes)`
that references an item id which no longer exists, when the
preference is normalized via `normalizePreferenceOrder, the missing`
id is retained (sorted to the end of the entry's list) rather than
silently deleted. The "unknown id last" rule is implemented by
`sortIdsByEntryOrder (entityOrdering.ts:28-37). Group members`
arrays are **not affected by this rule — **`normalizeStateForFrontend`
preserves existing group member order, and item-render/editor
paths may drop or log missing members separately (see spec 03).
