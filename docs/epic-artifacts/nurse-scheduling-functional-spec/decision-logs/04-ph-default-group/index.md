# Decision Log — Add `PH (Public Holidays) default date group`

## Context

The Singapore holiday import previously produced two editable date groups:
`WORKDAY (weekdays minus PH) and NON-WORKDAY (weekends ∪ PH). Rostering`
often needs to target *public holidays specifically (distinct from ordinary*
weekends), which `NON-WORKDAY conflates.`

## Settled decisions

- **New default date group ****`PH ("Public Holidays"), alongside `**`WORKDAY`
and `NON-WORKDAY. Seeded empty in the default state and`
populated/overwritten by the Singapore holiday import.
- **Membership = imported public holidays only. A date item is a **`PH`
member iff its date is in the imported public-holiday set — i.e. a gazetted
holiday. **Non-holiday weekends are excluded from **`PH. PH ⊆ NON-WORKDAY.`
- **Observed substitutes are included. Both actual holiday dates and their**
"(Observed)" substitute days are `PH members, consistent with how`
`NON-WORKDAY already treats them.`
- **`PH`**** is an ordinary editable group (not **`isAutoGenerated, not a`
reserved keyword), like `WORKDAY/NON-WORKDAY. It is created/overwritten`
by `replaceDateGroups at save time when import is effective.`
- **Scope boundary — no export column for ****`PH`****. The export layout still**
emits OFF-count columns only for `WORKDAY/NON-WORKDAY; PH was not wired`
into `schedulingExportConfig. Day-type classification`
(`getSingaporeDayType) is likewise unchanged (still WORKDAY | NON-WORKDAY);`
`PH is a membership group, not a calendar day-type.`
